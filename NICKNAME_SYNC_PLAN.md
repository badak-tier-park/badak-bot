# 유저 닉네임 강제 동기화 기능 아키텍처 (소규모 커뮤니티용 하이브리드)

본 문서는 트래픽 부하가 적은 소규모(50여 명) 디스코드 커뮤니티 환경에 맞춰, 
유저의 닉네임을 DB 정보와 강제로 일치시키는 동기화 기능의 구현 계획을 기록한 문서입니다.

## 핵심 기조
- 복잡한 실시간 청취 이벤트(`on_member_update`) 제외
- 관리자가 상호작용하는 즉시 닉네임이 바뀌는 **즉각적 피드백(UX)** 중시
- 어긋난 데이터를 매일 새벽 몰아서 교정하는 **가벼운 스케줄링(안전망)** 중시

---

## 1. 승인 시 눈앞에서 닉네임 즉시 변경 (즉각 반응)
관리자가 유저의 데이터를 바꿨을 때, 스케줄러가 돌 때까지 기다리지 않고 "눈앞에서 즉시" 닉네임이 바뀌는 쾌감을 줍니다.

- **적용 대상**: `cogs/admin.py` 내부 `NicknameApprovalView`, `RaceApprovalView` 등 관리자 처리 로직
- **작동 방식**: 
  - DB 업데이트 성공 직후 즉시 `await member.edit(nick="{nickname} / {race} / {tier}")` 실행.
  - 권한 문제(서버장 등 봇보다 등급이 높은 유저) 발생 시 `discord.Forbidden`을 캐치하여 봇 크래시 방지.

## 2. 가벼운 안전망을 위한 스케줄러 (배치 작업)
유저가 임의로 디스코드 프로필 설정에서 닉네임을 바꾸거나, 봇이 꺼져있을 때 누락된 에러를 매일 주기적으로 훑고 고쳐줍니다. 전체 50여 명이면 1초씩 쉬어도 50초 안에 끝나는 매우 가볍고 안전한 작업입니다.

- **적용 대상**: 신규 생성 파일 `cogs/schedule.py`
- **작동 방식**: 
  - `discord.ext.tasks` 루프를 사용해 지정된 시각(기본 04:00)에 작동.
  - DB `users` 전체를 순회하며 닉네임 비교 후 틀린 사람만 덮어쓰기. (디스코드 API 제한을 막기 위해 수정 건마다 `asyncio.sleep(1)` 대기 처리)

## 3. 관리자 제어 명령어 (`/설정`, `/닉네임수동동기화`)
운영 편의성을 위해 스케줄러 작동 시간 및 수동 강제 실행 명령어를 추가합니다.

- `/설정 동기화시각: 03:00` : `.env` 정보를 영구 업데이트 하고 백그라운드 태스크의 시간을 즉각 재세팅 (`config.py`와 연동)
- `/닉네임수동동기화` : 스케줄러 시간까지 기다리지 않고, 명령어를 친 즉시 `cogs/schedule.py`의 안전망 루틴을 한 바퀴 강제로 돌립니다.
